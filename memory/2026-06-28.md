## Дополнение: RLS fix — public_leaderboard view (2026-06-28, вечер)

### Проблема
Пользователь заметил, что в Supabase Table Editor видно 4 пользователя в `user_progress`, но в приложении:
- Leaderboard (/rating) — пустой или показывает только себя
- TeacherAnalytics ("Аналитика всех пользователей") — пустой или demo-данные
- UsersPage (/users) — показывает только 1 пользователя

### Причина
`user_progress` имеет RLS policy: `auth.uid() = user_id`. Это означает, что каждый авторизованный пользователь видит ТОЛЬКО свою собственную запись. Supabase Table Editor использует `service_role` key (админ) и обходит RLS — поэтому там видны все 4 записи. Клиентский код использует `anon` key с RLS — поэтому получает только 1 запись.

### Решение
1. Создан `public_leaderboard` view (security definer, owner = postgres) — bypasses RLS:
   ```sql
   CREATE OR REPLACE VIEW public_leaderboard AS
   SELECT user_id, user_stats, lesson_progress, task_stats, ... FROM user_progress;
   ```
2. Добавлен teacher policy: учителя могут видеть прогресс своих учеников через `teacher_student_links`.
3. SQL-функции `get_leaderboard()`, `get_all_user_progress()`, `get_all_users_basic()` (security definer) — fallback.
4. Обновлён код:
   - `syncSlice.ts` → `loadLeaderboard()` теперь `supabase.from('public_leaderboard')`
   - `teacherAnalyticsStore.ts` → `fetchAllUsers()` теперь `supabase.from('public_leaderboard')`
   - `UsersPage.tsx` → `loadUsers()` теперь `supabase.from('public_leaderboard')`
